1. OVERVIEW
================================================================================

Mezz is a dual-portal Flutter application for managing hostel mess operations.
It serves two user types:

  • STUDENTS  — Mark daily meals (Veg/Non-Veg/Absent), view expenditure,
                request meal changes or parcels, select billing plans, view
                receipts, and participate in a community feed.

  • VENDORS   — View daily headcounts, manage student billing, issue semester
                fees, handle unlock/parcel requests, clear bills with
                receipt generation, and configure settings.

Students link to vendors via a 6-digit vendor code. All data is stored in
Cloud Firestore. The UI follows a soft neumorphism design style with the
primary brand color #6C63FF (purple/indigo).

2. TECH STACK
================================================================================

  Framework:      Flutter (Dart SDK ^3.9.2)
  Backend:        Firebase (Firestore + Firebase Auth)
  Authentication: Firebase Authentication (email/password)
  Database:       Cloud Firestore (NoSQL document-based)
  Image Upload:   Cloudinary (unsigned upload preset)
  PDF Generation: pdf + printing packages (on-demand, client-side)
  Fonts:          Google Fonts (Poppins for headings, Manrope for body)
  State Mgmt:     setState + StreamBuilder (no external state management)

  Key Dependencies (pubspec.yaml):
  ─────────────────────────────────
  firebase_core, firebase_auth, cloud_firestore
  google_fonts, intl, http, image_picker, provider
  cached_network_image, pdf, printing, path_provider
  flutter_launcher_icons (dev dependency)

5. DATABASE SCHEMA (FIRESTORE)
================================================================================

All Firestore document IDs use email addresses (for students/vendors) or
auto-generated IDs (for subcollection items).

─── COLLECTION: students ──────────────────────────────────────────────

  Document ID: {student_email}
  Fields:
  ┌──────────────────┬──────────┬──────────────────────────────────┐
  │ Field            │ Type     │ Description                      │
  ├──────────────────┼──────────┼──────────────────────────────────┤
  │ email            │ string   │ Student's email address           │
  │ name             │ string   │ Full name                        │
  │ hostel_no        │ string   │ Hostel number (e.g., "5")        │
  │ profile_pic_url  │ string   │ Cloudinary URL for profile pic   │
  │ vendor_code      │ string   │ 6-digit code linking to vendor   │
  │ billing_status   │ string   │ 'active', 'paid', or 'pending'    │
  │ selected_plan    │ int      │ Plan ID: 1, 2, or 3             │
  │ role             │ string   │ Always 'student'                 │
  │ created_at       │ timestamp│ Account creation date            │
  └──────────────────┴──────────┴──────────────────────────────────┘

  Subcollection: students/{email}/vendor_data/{vendor_code}/

    Sub-subcollection: meals/{month}/days/{day}
    ┌──────────┬──────────┬────────────────────────────────────────┐
    │ Field    │ Type     │ Description                            │
    ├──────────┼──────────┼────────────────────────────────────────┤
    │ day      │ int      │ Day of month (1-31)                    │
    │ month    │ int      │ Month (1-12)                           │
    │ year     │ int      │ Year (e.g., 2026)                      │
    │ meals    │ array    │ [breakfast, lunch, dinner]              │
    │          │          │ Each: 0=Absent, 1=Veg, 2=Non-Veg       │
    └──────────┴──────────┴────────────────────────────────────────┘

    Sub-subcollection: billing/{month}
    ┌──────────────┬──────────┬────────────────────────────────────┐
    │ Field        │ Type     │ Description                        │
    ├──────────────┼──────────┼────────────────────────────────────┤
    │ month        │ int      │ Month number                       │
    │ monthly_exp  │ double   │ Total expenditure for the month    │
    │ total_year_exp│ double  │ Running yearly total               │
    │ paid_status  │ string   │ 'paid', 'pending', or 'partial'    │
    └──────────────┴──────────┴────────────────────────────────────┘

    Sub-subcollection: receipts/{receipt_id}
    ┌────────────────┬──────────┬──────────────────────────────────┐
    │ Field          │ Type     │ Description                      │
    ├────────────────┼──────────┼──────────────────────────────────┤
    │ id             │ string   │ Auto-generated doc ID            │
    │ receipt_code   │ string   │ Unique 8-char alphanumeric code  │
    │ expenditure    │ double   │ Total month expenditure           │
    │ plan_deduction │ double   │ Deducted from plan allowance     │
    │ final_cost     │ double   │ Amount owed after deductions     │
    │ buffer_used    │ double   │ Security deposit buffer used     │
    │ date           │ string   │ ISO 8601 date string             │
    │ student_email  │ string   │ Student's email                  │
    │ vendor_code    │ string   │ Vendor's 6-digit code            │
    │ month          │ int      │ Billing month                    │
    └────────────────┴──────────┴──────────────────────────────────┘

    Sub-subcollection: notifications/{notif_id}
    ┌──────────────┬──────────┬────────────────────────────────────┐
    │ Field        │ Type     │ Description                        │
    ├──────────────┼──────────┼────────────────────────────────────┤

6. DATA MODELS
================================================================================

All models are in lib/models/. Each has:
  - Constructor with named parameters
  - factory fromMap() — converts Firestore doc to model
  - toMap() — converts model to Firestore-compatible map
  - copyWith() — immutable update pattern

  StudentModel    — email, name, hostelNo, profilePicUrl, vendorCode,
                    billingStatus ('active'|'paid'|'pending'), selectedPlan (1-3)

  VendorModel     — email, name, uniqueCode (6-digit), menuConfig (map)

  MealDayModel    — day, month, year, meals (List<int> of 3 elements)
                    Each meal: 0=Absent, 1=Veg, 2=Non-Veg

  MonthBillingModel — month, monthlyExp (total ₹), totalYearExp,
                      paidStatus ('paid'|'pending'|'partial')

  ReceiptModel    — id, receiptCode (8-char), expenditure, planDeduction,
                    finalCost, bufferUsed, date, studentEmail, vendorCode, month
                    Static method: generateReceiptCode() — cryptographic random

  CommentModel    — id, authorEmail, authorName, text, imageUrl, likes, timestamp

7. SERVICES LAYER
================================================================================

All services are in lib/services/. They are stateless utility classes.

── auth_service.dart ──────────────────────────────────────────────────

  AuthService wraps Firebase Auth and Firestore user creation.
  
  Methods:
    getUserRole()     — Checks vendors then students collection, returns role
    signUpStudent()   — Creates Firebase Auth user + Firestore student doc
    signUpVendor()    — Creates Firebase Auth user + Firestore vendor doc
                        with auto-generated 6-digit vendor code
    signIn()          — Firebase Auth email/password sign-in
    signOut()         — Firebase Auth sign-out

  Role Detection: Checks if email exists in 'vendors' first, then 'students'.
  Vendor Code: Random 6-digit number generated via _generateVendorCode().


── firestore_service.dart (30+ methods) ───────────────────────────────

  The core data access layer. All Firestore reads/writes go through here.

  Student/Vendor CRUD:
    getStudent(), updateStudent(), getVendor(), updateVendor(), getVendorByCode()

  Meal Management:
    setMealDay()      — Write a day's meals to Firestore
    getMealDay()      — Read a single day's meals
    getMonthMeals()   — Read all days in a month

  Billing:
    setMonthBilling(), getMonthBilling()
    recalculateMonthlyExpenditure() — Sums all daily meal costs for the month,
                                      writes total to billing doc, returns total
    clearMonthBilling()             — Resets monthly_exp=0, paid_status='paid'

  Receipts:
    addReceipt()        — Save to student's receipts subcollection
    getReceipts()       — Read all student receipts
    addVendorReceipt()  — Save to vendor's receipts subcollection
    getVendorReceipts() — Read all vendor receipts

  Comments (Mess Feed):
    addComment(), commentsStream(), likeComment()

  Requests:
    sendRequest()       — Create unlock/parcel request in vendor's active_requests
    requestsStream()    — Real-time stream of vendor's active requests

  Student Search:
    getStudentsByVendorCode() — All students linked to a vendor
    searchStudents()          — Filter by email or hostel number

  Headcounts:
    getMealHeadcounts()  — Counts veg/non-veg/absent for each meal on a given day

  Notifications:
    notificationsStream()    — Real-time stream from student's notification subcollection
    addStudentNotification() — Write notification to student (with push-style fields)

  Pending Bills:
    setPendingBill()         — Create/update pending bill on vendor side
    deletePendingBill()      — Remove pending bill by student+month+type
    pendingBillsStream()     — Real-time stream of vendor's pending bills
    getPendingBillsForStudent() — Query pending bills for specific student


── meal_logic.dart ────────────────────────────────────────────────────

  Pure Dart logic — no Firebase dependency.

  Enums: MealType (breakfast, lunch, dinner), MealState (absent, veg, nonVeg)

  Key Methods:
    isMealLocked()       — Determines if a meal can be changed
                           Past days: always locked
                           Today: locked during (start-60min) to (end+30min)
                           Future: always unlocked

    getTodayMealStatuses() — Returns lock status for all 3 meals

    cycleMealState()     — Cycles: Absent(0) → Veg(1) → NonVeg(2) → Absent(0)

    getMealCost()        — Per-meal cost lookup from MealSchedule constants
    getDailyCost()       — Sum of 3 meals for one day


── billing_logic.dart ─────────────────────────────────────────────────

  Pure Dart logic for semester billing calculations.

  BillingResult holds: monthlyExpense, planAllowance, overspend,
                       bufferUsed, pendingBill, remainingBuffer

  Key Methods:
    getMonthlyAllowance() — Plan lookup: Plan1=₹2000, Plan2=₹2500, Plan3=₹3000
    calculateMonthBilling() — Core formula:
        overspend = max(0, monthlyExpense - planAllowance)
        If overspend > 0:
          If buffer >= overspend → buffer covers it
          Else → partial buffer + remaining becomes pendingBill

    getPlanLabel(), getAllPlans() — Display helpers


── cloudinary_service.dart ────────────────────────────────────────────

  Handles profile picture uploads to Cloudinary CDN.
  Uses unsigned upload preset via HTTP POST to Cloudinary's upload API.
  Returns the secure_url of the uploaded image.


── receipt_pdf_service.dart ───────────────────────────────────────────

  Generates styled PDF receipts on-demand (NOT stored in DB or cloud).
  Uses the 'pdf' package to build a professional receipt with:
    - Header with "MEZZ" branding
    - Student and vendor details
    - Itemized billing breakdown
    - Unique 8-character receipt code
    - Date and signature section
  
  Uses the 'printing' package to trigger system share/save dialog.

── Vendor Billing (vendor_billing.dart) ────────────────────────────────

  Two types of content in the list:

  1. Semester Fee Cards (red, at top):
     - Shows unpaid semester fees from pending_bills collection
     - "Mark as Paid" button → sets student billing_status='paid',
       deletes pending bill, notifies student

  2. Monthly Billing Cards:
     - Current month: "In Progress" badge, NO clear button
     - Past months (unpaid): "Pending" badge + "Clear Bill" button
     - Clearing a bill:
       a. Generates unique 8-char receipt code
       b. Calculates billing via BillingLogic
       c. Creates ReceiptModel → saves to student & vendor receipts
       d. Clears student's month billing (monthly_exp=0, paid_status='paid')
       e. Deletes monthly pending bill from vendor DB
       f. Sends notification to student with receipt code
       g. Offers PDF download in snackbar

  3. "Issue New Semester Fee" button:
     - Iterates all linked students
     - For 'paid' students: sends notification, sets billing_status='pending'
     - For 'pending' students: re-writes pending bill silently
     - Creates semester_fee type pending bill for each student

  Filter chips: All / Unpaid


── Vendor Notifications (vendor_notifications.dart) ────────────────────

  Real-time stream of active_requests from vendor's subcollection.
  Each request shows:
    - Student name, meal type, reason

    - Meal card is read-only

12. FEATURE: BILLING & PLANS
================================================================================

  BILLING FLOW:

  ┌──────────────────────────────────────────────────────────┐
  │              DAILY MEAL TRACKING                          │
  │                                                          │
  │  Student marks meals → getMealCost() per meal → sum      │
  │  Written to: meals/{month}/days/{day}                    │
  │                                                          │
  │  After each change: recalculateMonthlyExpenditure()      │
  │  Sums all days → updates billing/{month}.monthly_exp     │
  │  Also syncs to vendor's pending_bills                    │
  └──────────────────────────┬───────────────────────────────┘
                             │
                             ▼
  ┌──────────────────────────────────────────────────────────┐
  │              MONTH-END BILLING (Vendor clears bill)      │
  │                                                          │
  │  calculateMonthBilling(expense, planId, buffer)          │
  │                                                          │
  │  planAllowance = Plan amount (₹2000/₹2500/₹3000)        │
  │  overspend = max(0, expense - planAllowance)             │
  │                                                          │
  │  If overspend > 0:                                       │
  │    If buffer ≥ overspend → buffer absorbs                │
  │    Else → buffer used up, rest = pendingBill             │
  │                                                          │
  │  Receipt generated with unique 8-char code               │
  │  Student notified, billing cleared                       │
  └──────────────────────────────────────────────────────────┘

  EXAMPLE:
    Student on Plan 1 (₹2,000/mo), monthly expense = ₹2,800
    Buffer remaining = ₹500

    overspend = ₹2,800 - ₹2,000 = ₹800
    buffer covers ₹500, pending bill = ₹300
    remainingBuffer = ₹0

16. FEATURE: REQUESTS (UNLOCK & PARCEL)
================================================================================

  UNLOCK REQUEST:
    Trigger: Student taps "Request" on a locked meal that is in Absent (0) state
    Contains: student email, meal label, reason, type='unlock'
    Vendor action:
      - Approve: Sets the meal to Veg (1) in student's Firestore data,
                 recalculates expenditure, sends notification
      - Reject: Sends rejection notification

  PARCEL REQUEST:
    Trigger: Student taps "Parcel" on a meal in Veg (1) or Non-Veg (2) state
    Contains: student email, meal label, reason, recipient name, type='parcel'
    Vendor action:
      - Approve: Marks as handled, sends notification to student
      - Reject: Sends rejection notification

  TIME BLOCKING:
    Both request types are blocked after meal end time + 30 minutes.
    The button simply disappears from the UI after this cutoff.

20. KEY BUSINESS RULES
================================================================================

  1. Meal changes are only allowed for future days, or today if the meal
     is outside its lock window (1 hour before to 30 min after).

  2. Student meal state cycles: Absent → Veg → Non-Veg → Absent.
     Double-tap to change.

  3. Every meal change triggers recalculateMonthlyExpenditure() which
     sums ALL meal costs for the month and syncs to vendor's pending bills.

  4. Semester fee is ₹13,500 total (security deposit included).
     Vendor issues it, student picks a plan, vendor confirms payment.

  5. Plans only affect the monthly deduction allowance:
     Plan 1: ₹2,000/mo    Plan 2: ₹2,500/mo    Plan 3: ₹3,000/mo

  6. If monthly expense exceeds plan allowance, the overspend is first
     covered by the security deposit buffer, then becomes a pending bill.

  7. Vendors can only clear bills for PAST months (not the current month).

  8. Each cleared bill generates a unique 8-character receipt code shared
     with both student and vendor.

  9. PDFs are generated on-demand from receipt data — never stored.

  10. Request/parcel buttons disappear after meal end time + 30 minutes.

  11. When a student first links a vendor code, a semester fee pending bill
      is automatically created.

21. HOW TO MODIFY COMMON THINGS
================================================================================

  CHANGE MEAL PRICES:
    Edit: lib/constants/constants.dart → MealSchedule class
    Fields: breakfastPrice, lunchVegPrice, lunchNonVegPrice, etc.

  CHANGE MEAL TIMES:
    Edit: lib/constants/constants.dart → MealSchedule class
    Fields: breakfastStart/End, lunchStart/End, dinnerStart/End (24-hour)

  CHANGE LOCK WINDOW:
    Edit: lib/constants/constants.dart → MealSchedule class
    Fields: lockBeforeMinutes (default: 60), lockAfterMinutes (default: 30)

  CHANGE BILLING PLANS:
    Edit: lib/constants/constants.dart → BillingConstants.plans
    Each plan has: amount (monthly ₹), months (duration), label (display)

  CHANGE SEMESTER FEE:
    Edit: lib/constants/constants.dart → BillingConstants.baseFee
    Currently ₹13,500 (includes security deposit)

  CHANGE APP COLORS:
    Edit: lib/constants/constants.dart → AppColors class
    Primary brand: primary (#6C63FF), primaryDark (#4A42E8)

  CHANGE APP LOGO:
    Replace: assets/images/mezz_logo.png
    Run: dart run flutter_launcher_icons
    (Also update pubspec.yaml adaptive_icon_background color if needed)

  ADD A NEW FIRESTORE FIELD:
    1. Add field to relevant model in lib/models/
    2. Update fromMap() and toMap() in the model
    3. Update relevant methods in firestore_service.dart
    4. Update UI screens that display/edit the field

  ADD A NEW SCREEN:
    1. Create file in lib/screens/{portal}/
    2. Add tab in the shell file (student_shell.dart or vendor_shell.dart)
    3. Add navigation item in floating_bottom_nav.dart

  CHANGE CLOUDINARY CONFIG:
    Edit: lib/services/cloudinary_service.dart
    Fields: cloudName, uploadPreset
